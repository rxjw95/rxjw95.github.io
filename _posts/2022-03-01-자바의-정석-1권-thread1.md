## 프로세스

프로세스는 일반적으로 CPU에 의해 처리되는 프로그램을 의미한다. 또한, 이 프로세스는 단순히 한마디로 정의할 수 있는게 아니라 다양한 형태로 정의될 수 있다.

프로세스는 다음과 같이 여러 형태로 정의할 수 있다.

-   PCB를 가진 프로그램
-   실기억장치에 저장된 프로그램
-   프로세서가 할당되는 실체로서, 디스패치가 가능한 단위
-   프로시저가 활동 중인 것
-   비동기적 행위를 일으키는 주체
-   지정된 결과를 얻기 위한 일련의 계통적 동작
-   목적 또는 결과에 따라 발생되는 사건들의 과정
-   운영체제가 관리하는 실행 단위

부가적으로 설명을 추가하자면,

![메모리 구조](https://user-images.githubusercontent.com/62179353/176123539-0f19b6df-3259-4662-9037-7cbff65796ed.png)

메모리에 적재된 프로세스는 각각 독립된 메모리 영역(text, data, stack, heap의 구조)을 할당받는다. 각 프로세스는 별도의 주소 공간에서 실행되며, 한 프로세스는 다른 프로세스의 변수나 자료구조에 접근할 수 없다.

한 프로세스가 다른 프로세스의 자원에 접근하려면 프로세스 간의 통신(IPC, inter-process communication)을 사용해야 한다.


프로세스 내에 각 독립된 영역을 간단히 설명하면 다음과 같다.

-   text : 컴파일된 소스 코드가 저장되는 영역
-   data: 전역 변수/초기화된 데이터가 저장되는 영역
-   stack: 임시 데이터(함수 호출, 로컬 변수 등)가 저장되는 영역
-   heap: 코드에서 동적으로 생성되는 데이터가 저장되는 영역

이때, 우리는 프로세스의 생성, 생명 주기, PCB와 메모리 등에 대한 대체적인 자료구조에 대해서 공부를 한다면 더욱 쉽게 이해할 수 있다.

해당 내용에 대한 공부는 공룡 책이나 공룡 책을 정리한 블로그들을 찾아보면 금방 공부할 수 있기 때문에 따로 작성하지 않는다.


## 스레드

스레드를 설명하기 위해선 프로세스가 반드시 언급이 되어야 한다. 기본적으로 프로세스당 최소 1개의 스레드(메인 스레드)를 가지고 있기 때문이다.


스레드의 정의는 다음과 같이 정의할 수 있다.

-   프로세스 내에서 실행되는 여러 흐름의 단위
-   프로세스의 특정한 수행 경로
-   프로세스가 할당받은 자원을 이용하는 실행의 단위

![프로세스와 스레드](https://user-images.githubusercontent.com/62179353/176123674-6d739ba2-12a5-4365-8b42-dee58a3f7f52.png)

스레드는 프로세스 내에서 각각의 stack 영역만 따로 할당받고 text, data, heap 영역은 공유한다.


스레드는 한 프로세스 내에서 동작되는 여러 실행의 흐름으로, 프로세스 내의 주소 공간이나 자원들(힙 공간 등)을 같은 프로세스 내에 스레드끼리 공유하면서 실행된다.

여기서 스레드 개념의 탄생 배경과 스레드는 왜 각각의 stack 영역만을 가지는지에 대해서도 공부하면 좋다.

## 멀티 태스킹(프로세스)과 멀티 스레드

멀티 프로세스 그리고 멀티 스레드를 사용할 때의 장점과 단점으로 비교해보자.

### 멀티 프로세스

하나의 응용 프로그램을 여러 개의 프로세스로 구성하여 각 프로세스가 하나의 작업을 처리하도록 하는 것이다.

**장점**

-   여러 개의 자식 프로세스 중 하나에 문제가 발생하면 그 자식 프로세스만 죽는 것 이상으로 다른 영향이 확산되지 않기 때문에 안정성이 높다.
-   메모리 침범에 대한 문제를 OS 차원에서 해결해 준다.

**단점**

-   프로세스는 각각의 독립된 메모리 영역을 할당받는 것이 CPU와 메모리 자원을 멀티 스레드보다 더 많이 사용하게 된다.
-   Context Swithching에서의 오버헤드가 멀티 스레드보다 더 크다.
-   프로세스 간 통신을 하기 위해선 복잡한 통신 기법이 필요하다.

**프로세스의 컨텍스트 스위칭**

-   다른 프로세스로의 메모리 전환이 이루어지며 그에 따른 메모리 할당 과정이 필요하다.
-   레지스터 상태 전환이 일어난다.
-   새 메모리 공간의 할당은 TLB(Transtation Lookaside Buffer) 플러싱이 일어난다.
-   TLB를 이용하여 가상 메모리 공간을 할당 속도를 높이는데(물리<->가상 매핑 캐시 역할) TLB 플러싱이 일어나면 다시 메모리를 읽고 매핑해야 한다.
-   캐싱의 의미가 없다.공유하는 데이터가 없으므로(독립적인 메모리 공간을 사용) 캐시가 쌓아놓은 데이터들이 무의미해지고, 새로 캐시를 저장해야 한다.
-   캐시는 CPU와 메인메모리 사이에 위치하며 CPU에서 한번 이상 읽어들인 메모리의 데이터를 저장하고 있다가, CPU가 다시 그 메모리에 저장된 데이터를 요구할 때, 메인 메모리를 통하지 않고 데이터를 전달해 주는 용도이다.

정리하면,

-   가상 메모리 공간이 유지되지 않기 때문에, 운영체제에서 메모리 할당 시간이 필요하다. (메모리 할당은 굉장히 오버헤드가 큰 작업)
-   캐싱의 의미가 없어진다.

**멀티 스레드의 컨텍스트 스위칭**

-   한 프로세스 내부에서 데이터를 공유하기 때문에 메모리 전환 과정이 필요 없다.
-   캐싱 효율 좋다 (메모리를 공유하고 있으니). 공유 메모리라면 메모리 접근 시 지역성 효율이 높다.

### 멀티 스레드


하나의 응용 프로그램을 여러 스레드를 구성해 각 스레드가 하나의 작업을 처리하는 것이다.


**장점**

-   시스템 자원 소모 감소(자원의 효율성 증대) : 프로세스를 생성하여 자원을 할당하는 시스템 콜이 줄어들어 자원을 효율적으로 관리
-   시스템 처리량 증가(처리 비용 감소) : 스레드 사이의 작업량이 작아 Context Switching이 빠르다.
-   간단한 통신 방법으로 인한 프로그램 응답 시간 단축 : 스레드는 프로세스 내의 Stack 영역을 제외한 모든 메모리를 공유하기 때문에 통신의 부담이 적다.


**단점**

-   스레드 하나가 프로세스 내 자원을 망쳐버린다면 모든 프로세스가 종료될 수 있다.
-   자원을 공유하기 때문에 필연적으로 **동기화 문제**가 발생할 수 밖에 없다.

---

다양한 글을 읽어보면서 무엇이 더 좋은 것일까, 멀티 스레드가 더 좋은 이유라고 적혀있는데, 나는 무엇이 더 좋고 나쁘고의 정답은 없다고 생각한다.

어떤 시스템, 애플리케이션을 개발할 것인지 그리고 중요하게 생각하는 점(리소스, 성능, 안정성 등)들을 적절히 분석하고 타협해서 가장 적합한 것을 선택해야 한다.

대표적으로 예전에 인터넷 익스플로러와 구글 크롬의 멀티 탭 기능을 사용할 때 멀티 스레드와 멀티 프로세스로 구분되는데,

인터넷 익스플로러는 멀티 탭을 띄울 때 스레드로 생성이 되며, 구글 크롬은 멀티 탭을 띄울 때 프로세스로 생성이 된다.

각각의 장단점은 있지만 인터넷 익스플로러는 멀티탭 기능을 개발할 때 리소스에 중점을, 크롬은 안정성(한 탭이 다른 탭에 영향이 없게 하도록 하기 위해서)에 중점을 두었기 때문이지 않을까 한다.
